# Plan

## 현재 목표

- 로컬 SSH 공개키를 원격 서버에 안전하게 등록하고, 등록된 키를 바로 사용할 수 있도록 로컬 SSH config까지 자동 반영한다.
- GUI, 로컬 키 관리, 원격 SSH 처리, 로컬 SSH config 처리의 역할 분리를 유지하면서 변경 범위를 좁게 관리한다.

## 최근 완료

- 원격 `authorized_keys` 등록 성공 또는 이미 등록된 키일 때 로컬 `~/.ssh/config` 자동 반영을 추가했다.
- `Host`, `HostName`, `Port`, `User`, `IdentityFile`, `IdentitiesOnly`, `PreferredAuthentications` 항목을 Host별로 추가/갱신한다.
- 로컬 config 반영 결과를 `added`, `updated`, `unchanged`로 사용자 메시지에 함께 표시한다.
- 프로젝트 버전을 `0.2.0`으로 올렸다.

## 가까운 작업 후보

- `Host` 별칭을 서버 주소와 별도로 입력할 수 있게 할지 검토한다.
- 키 생성/삭제 로직에 대한 테스트를 추가해 회귀 가능성을 줄인다.
- `README.md`와 빌드 절차를 기준으로 플랫폼별 배포 체크리스트를 정리한다.
- `known_hosts` 자동 등록 흐름에 대한 사용자 안내 문구를 더 명확히 다듬을지 검토한다.

## 구조상 유지할 원칙

- UI는 `ui/`에 두고, SSH/파일 처리 로직은 서비스 및 하위 모듈로 분리한다.
- 원격 서버 조작은 `ssh/remote.py`에 집중시켜 예외와 권한 처리 위치를 일원화한다.
- 로컬 OpenSSH config 조작은 `ssh/local_config.py`에 집중시킨다.
- 사용자에게 보이는 메시지 포맷은 `services/register.py`에서 정리한다.

## 중기 검토 항목

- 서버 목록 저장 기능을 넣을지 여부
- 다중 서버 동시 등록 지원 여부
- SSH 키 기반 관리자 로그인 지원 여부
- 원격 키 제거 기능 추가 여부

## 리스크

- 플랫폼별 `ssh-keygen` 가용성 차이로 키 생성 UX가 달라질 수 있다.
- 미등록 호스트 자동 신뢰 정책은 편의성이 높지만 보안 설명이 충분히 필요하다.
- 기존 `~/.ssh/config`의 복잡한 `Host`/`Match` 구성은 실제 사용자 파일로 추가 검증이 필요하다.
- GUI 중심 앱이라 수동 테스트 의존도가 높다.
